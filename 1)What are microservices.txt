1)What are microservices?
Microservices is an architecture where the application is exposed as loosely coupled services that can be independently developed, deployed, and maintained. Each service exposed is referred to as Microservice. Each service performs a unique function.

Speciality of this architecture is that polyglot architecture is supported.  For example, if a team is working on one of the microservice using Java, Spring Boot, and MySQL, another team can work on another microservice using Python, Node JS, and NoSQL. 

Different microservices can use a different version of the same programming language.
Different microservices can use different programming languages.
Different microservices can use different architectures as well.


2)Why microservices?
In the case of monolith applications, there are several problems like
Same code base for presentation, business layer, and data access layer. Application is deployed as a single unit.
Complex to maintain and scalability is an issue.
Microservice solves the above problems. 
Microservices are ideal when a monolith or a legacy application needs to be modernized. 
For new software development, if the key business drivers are to reduce time to market, scalable better software, lower costs, faster development, or cloud-native development, microservices are ideal.
Each service is independent and gives the flexibility to choose the programming language, database, and/or architecture.
Distinct services can be developed, deployed, and maintained independently.

3)Pros and cons of Microservices Architecture
 Pros of Microservice Architecture
1) Freedom to use different technologies
2) Each microservices focuses on single capability
3) Supports individual deployable units
4) Allow frequent software releases
5) Ensures security of each service
6) Multiple services are parallelly developed and deployed

 Cons of Microservice Architecture
1) Management of a large number of services is difficult.
2) Communication between microservices is complex.
3) Increased efforts for configuration and other operations
4) Difficult to maintain transaction safety and data boundaries
5) Due to the decentralized nature of microservices, more microservices will mean more resources hence high Investment
6) Debugging of problems is harder unless the right instrumentation is followed during design and development.
7) Microservices will need a large team size with the right mix of experience in design, development, automation, deployments, tools, and testing.

7)How do microservices communicate with each other?
Synchronous Communication
Services make direct requests and wait for responses immediately.
REST (Representational State Transfer)
Most common approach - Uses HTTP/HTTPS
gRPC (Google Remote Procedure Call)
Modern, high-performance approach - Uses HTTP/2 and Protocol Buffers

Asynchronous Communication (Event-Driven)
Services don't wait for immediate responses. Communication happens through events and messages.
Message Brokers/Queues like RabbitMQ, Kafka, AWS SQS
Services publish messages to a broker, and other services consume them.

10)Explain the role of API Gateway in microservices
Service Aggregation
The API Gateway aggregates many microservices into a single API endpoint. This reduces the number of requests made by the client.
Protocol Translation
The API gateway can translate many protocols like HTTP, Websockets and GRPC. This helps the client to choose the communication protocol without changing the microservices themselves.
Requests Routing
It routes the requests to different microservices based on URL, headers.
Authorization and Authentication
It handles a and a based on API keys, Oauth tokens, JWT tokens.
Load Balancing
It distributes the requests to multiple instances of the microservices.
Rate Limiting
It also enforces rate limiting to prevent the client making too many requests in short period of time.

11)How would you handle a microservice failure?
1)Retry Mechanism:
Failures are temporary. Instead of declaring a services completely failing we can use retry mechanism to reattempt the request after a short delay. Ex: Database Connection Timeouts. 
2)Circuit Breaker: Instead of repeatedly trying retry it will cause overhead for the services. We can use circuit breaker pattern which stops sending /blocks requests to the failing service and reroutes to other instances.
3)Timeouts:
It defines maximum time a service can wait for a response from other service so that if one service fails the other service does not keeps on waiting causing lag.
4)Message Queue for Asynchronous Processing: Use Message Queue like RabbitMQ, Kafka for decoupling the messages.
5)Logging and Monitoring Tools: Use them to detect issues early preventing them from failure.

12)Explain Circuit Breaker Pattern
The Circuit Breaker pattern is like a safety switch for microservices. Suppose your online store depends on a payment service. If that service starts failing repeatedly, the circuit breaker “trips” and stops further calls for a while, preventing cascades and giving the service time to recover.
It has got three states:
Closed State
In the Closed state, the circuit breaker operates normally, allowing requests to flow through between services.
During this phase, the circuit breaker monitors the health of the downstream service by collecting and analyzing metrics such as response times, error rates, or timeouts.
Open State
When the monitored metrics breach predetermined thresholds, signaling potential issues with the downstream service, the circuit breaker transitions to the Open state.
In the Open state, the circuit breaker immediately stops forwarding requests to the failing service, effectively isolating it.
Half-Open State
After a specified timeout period the CB will transition to half-open state. It will send limited number of trial requests to the downstream service. If the response is got indicating service recovery then CB will again go back to closed state. Or else if the trail request fail then it will be further evaluated.
Some of the tools which implement CB pattern are
1)Netflix Hystrix:
Hystrix is an open-source Java library that provides an implementation of the circuit breaker pattern to handle faults tolerance and latency in distributed systems and microservices architectures.
2. Resilience4j:
Resilience4j is a lightweight, easy-to-use library, designed with functional programming approach.
Key features of Resilience4j include:
a)Retry
b)Rate Limiting
c)Fallback: It allows for designing fallback responses if any failure occurs. 
d)Functional Programming

17)What is event driven architecture in microservices?
Event-driven microservices are an architectural style where independent services communicate asynchronously by producing and consuming events (state changes) rather than direct API calls.
Key Components & Concepts:
Event Producers: Services that detect and emit events (e.g., "OrderPlaced").
Event Consumers: Services that listen for and react to events, acting independently.
Message Broker (Event Bus): Middleware (like Apache Kafka, RabbitMQ, or AWS SNS/SQS) that manages the ingestion, storage, and distribution of events.
Asynchronous Communication: Services do not block waiting for a response, increasing throughput.
Eventual Consistency: Data across services may not update instantly, but will become consistent eventually, allowing high performance.

26)Database per service- why is it recommended?
1.Decoupling Services
Each microservice operates independently, without depending on a shared database schema. This decoupling ensures that changes to the database of one service don’t affect the others. 
2.Improved Scalability
Since each microservice manages its own database, you can scale the services independently.
3.Fault Isolation & Resilience: Database failures in one service are contained, preventing a "domino effect" that could crash the entire application.
4.Security: Enhanced security through granular access controls, as each service only has permission to access its own data.

30)How do you implement idempotent REST APIs?
Idempotency is a property of API design that ensures that making the same request multiple times produces the same result as making it once.
Consider an API endpoint that is designed to sign up a new user account in a web application. If this API is idempotent, it means that no matter how many times the API is called with the same input data (e.g., the same email and password), it will create the user account only once, and any subsequent invocations will have no further effect.
Use Cases:
Payment Processing: Idempotent APIs prevent double charging when processing payments.
Order Processing: Idempotent APIs in e-commerce platforms avoid duplicate orders.
File Uploads: Idempotent APIs for file uploads prevent unnecessary duplication.
How to implement idempotent Rest APIS
Assign unique identifiers: Use UUIDs or other unique identifiers for each request to track and identify requests.
Idempotent HTTP methods: Design APIs using idempotent HTTP methods like GET, PUT, and DELETE. These methods ensure that multiple identical requests have the same effect as a single request.
Expiration time for idempotency keys: Set a reasonable expiration time
